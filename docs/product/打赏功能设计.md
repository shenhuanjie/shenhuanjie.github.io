## 一、设计背景

### 1.1 目的

为读者提供一种轻量级的支持创作者方式，通过打赏功能让读者可以自愿资助优质内容的创作。

### 1.2 打赏场景

| 场景 | 描述 |
|------|------|
| 文章受益 | 读者从文章中获得了实际帮助 |
| 技术解决 | 文章解决了读者的问题 |
| 情感认同 | 读者认可创作者的付出 |
| 鼓励创作 | 读者希望看到更多优质内容 |

## 二、打赏方式设计

### 2.1 主流打赏方式对比

| 方式 | 优点 | 缺点 | 推荐度 |
|------|------|------|--------|
| 微信支付 | 普及率高、操作便捷 | 个人收款限额 | ★★★★★ |
| 支付宝 | 普及率高 | 需跳转 App | ★★★★☆ |
| 微信赞赏码 | 专用于赞赏场景 | 需保存图片 | ★★★★★ |
| 支付宝收款码 | 专用于收款 | 需保存图片 | ★★★★☆ |
| PayPal | 国际化 | 国内使用率低 | ★★☆☆☆ |

### 2.2 推荐方案：微信/支付宝双码

**理由：**
1. 覆盖国内 95% 以上用户
2. 操作流程简单，30 秒内完成
3. 无平台手续费（个人转账）
4. 赞赏码形式更适合"赞助"场景

## 五、打赏流程设计

### 5.1 用户操作流程

```
用户点击"微信打赏"
        ↓
弹出打赏面板，显示微信二维码
        ↓
用户截图 → 打开微信 → 扫码支付
        ↓
返回博客页面（可选择是否留言）
        ↓
显示"感谢支持"提示
        ↓
打赏记录入库
```

### 5.2 打赏反馈机制

| 打赏金额 | 反馈方式 |
|----------|----------|
| 任意金额 | 评论区感谢回复 |
| 30 元以上 | 邮件感谢 + 付费内容优惠券 |
| 100 元以上 | 加入"特别感谢"名单 |
| 年度累计前10 | 文章底部"赞助商"标识 |

## 七、技术实现方案

### 7.1 前端组件

### 7.2 打赏记录后端

```javascript
// 打赏记录接口
app.post('/api/reward/record', async (req, res) => {
  const { amount, method, message, anonymous } = req.body;

  // 记录打赏（需人工确认转账）
  const record = await db.rewardRecords.create({
    amount,
    method, // wechat | alipay
    message: message || '',
    anonymous: anonymous !== false,
    status: 'pending', // pending | confirmed
    createdAt: new Date()
  });

  // 发送通知给博主
  await sendNotification(`收到打赏 ¥${amount}`);

  res.json({ success: true, recordId: record.id });
});
```

## 八、打赏二维码生成

### 8.1 二维码获取方式

| 方式 | 操作步骤 |
|------|----------|
| 微信收款码 | 微信 → 我 → 服务 → 收付款 → 收款码 |
| 支付宝收款码 | 支付宝 → 收钱 → 领取收钱码 |

### 8.2 二维码优化

- 尺寸：400x400 像素
- 格式：PNG 或 WebP
- 文件大小：< 100KB
- 容错率：H 级（30%）

### 8.3 定期更换

- 建议每 3 个月更换一次二维码
- 避免二维码过期导致无法收款
- 更换后及时更新图片

## 十、风险控制

### 10.1 安全风险

| 风险 | 应对措施 |
|------|----------|
| 二维码被盗用 | 定期更换 + 水印处理 |
| 虚假转账 | 人工审核确认 |
| 账户异常 | 备用收款账户 |

### 10.2 隐私风险

| 风险 | 应对措施 |
|------|----------|
| 用户信息泄露 | 默认匿名展示 |
| 金额暴露 | 可选择隐藏金额 |
| 恶意骚扰 | 评论区审核机制 |
